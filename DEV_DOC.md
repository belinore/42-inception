# Developer Documentation

## Project Goal

This project builds a complete WordPress stack with Docker Compose:

```text
nginx -> WordPress/PHP-FPM -> MariaDB
```

The implementation uses one custom Docker image per service. The images are built from Debian, not from ready-made nginx, WordPress, or MariaDB Docker images.

## Set Up From Scratch

Required tools on the VM:

- Docker
- Docker Compose
- `make`
- `openssl`

Required project configuration files:

```text
Makefile
srcs/docker-compose.yml
srcs/.env
srcs/requirements/tools/setup.sh
srcs/requirements/mariadb/Dockerfile
srcs/requirements/mariadb/conf/50-server.cnf
srcs/requirements/mariadb/tools/init.sh
srcs/requirements/nginx/Dockerfile
srcs/requirements/nginx/conf/default.conf
srcs/requirements/wordpress/Dockerfile
srcs/requirements/wordpress/conf/www.conf
srcs/requirements/wordpress/tools/init.sh
```

Required local secret files:

```text
secrets/MARIADB_ROOT_PASSWORD
secrets/MARIADB_PASSWORD
secrets/WP_ADMIN_PASSWORD
secrets/WP_USER_PASSWORD
secrets/SSL_CERTIFICATE
secrets/TLS_PRIVATE_KEY
```

Setup order:

1. Confirm Docker and Docker Compose are installed.
2. Confirm `srcs/.env` contains the non-secret project settings.
3. Optional - create the required password files under `secrets/` (otherwise they will be auto-generated).
4. Start the project with `make` - this will generate secrets, SSL keys, build and launch the containers.

## Services

### nginx

Path:

```text
srcs/requirements/nginx
```

Purpose:

- public HTTPS entrypoint
- listens on port `443`
- uses TLSv1.2 and TLSv1.3 only
- serves WordPress files from the shared WordPress volume
- forwards PHP requests to `wordpress:9000`

Important files:

```text
srcs/requirements/nginx/Dockerfile
srcs/requirements/nginx/conf/default.conf
```

TLS files are mounted as Docker secrets:

```text
/run/secrets/ssl_certificate
/run/secrets/tls_private_key
```

### WordPress

Path:

```text
srcs/requirements/wordpress
```

Purpose:

- installs WordPress
- runs PHP-FPM in the foreground
- creates `wp-config.php` with `wp-cli`
- installs the WordPress site
- creates the administrator user and a second author user

Important files:

```text
srcs/requirements/wordpress/Dockerfile
srcs/requirements/wordpress/conf/www.conf
srcs/requirements/wordpress/tools/init.sh
```

The active runtime `wp-config.php` is generated by `wp-cli` inside the WordPress volume.

### MariaDB

Path:

```text
srcs/requirements/mariadb
```

Purpose:

- initializes the MariaDB data directory if needed
- creates the WordPress database
- creates the WordPress database user
- grants privileges on the WordPress database
- runs `mariadbd` in the foreground

Important files:

```text
srcs/requirements/mariadb/Dockerfile
srcs/requirements/mariadb/conf/50-server.cnf
srcs/requirements/mariadb/tools/init.sh
```

MariaDB listens on `0.0.0.0:3306` inside the Docker network so WordPress can connect to it.


```text

```

Network:

```text

```

Named volumes:

```text
mariadb_data

```

Only nginx publishes a host port:

```yaml
ports:
  - "443:443"
```

WordPress and MariaDB are reachable only through the Docker network.

## Secrets

Secrets are stored as local files under:

```text
secrets/
```

Required files:

```text
secrets/MARIADB_ROOT_PASSWORD
secrets/MARIADB_PASSWORD
secrets/WP_ADMIN_PASSWORD
secrets/WP_USER_PASSWORD
secrets/SSL_CERTIFICATE
secrets/TLS_PRIVATE_KEY
```

These files are ignored by git through:

```text
secrets/.gitignore
```

Create password secrets with `printf` so accidental newlines are not added:

```sh
mkdir -p secrets
printf '%s' 'choose_a_root_db_password' > secrets/MARIADB_ROOT_PASSWORD
printf '%s' 'choose_a_wp_db_password' > secrets/MARIADB_PASSWORD
printf '%s' 'choose_a_wp_admin_password' > secrets/WP_ADMIN_PASSWORD
printf '%s' 'choose_a_wp_author_password' > secrets/WP_USER_PASSWORD
```

For an existing installation, changing these files alone may not update the stored MariaDB or WordPress credentials. Use `make fclean_data` for a fresh install with new passwords.

The SSL certificate will be auto-generated or can be created manually with:

```sh
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
	-keyout secrets/TLS_PRIVATE_KEY \
	-out secrets/SSL_CERTIFICATE \
	-subj "/CN=belinore.42.fr"
```

## Build And Run

From the repository root:

```sh
make
```

This runs:

- `./srcs/requirements/tools/setup.sh` to create the configured data directories and generate secrets.
- `docker compose -f srcs/docker-compose.yml up --build`

Stop containers:

```sh
make down
```

Rebuild images:

```sh
make build
```

Show containers:

```sh
make ps
```

Show logs:

```sh
make logs
```

Stop and remove containers:

```sh
make clean
```

Stop containers and remove Compose volumes:

```sh
make fclean
```

Remove Compose volumes, local data, and generated secrets:

```sh
make fclean_data
```

## Validation Commands

Validate Compose:

```sh
docker compose -f srcs/docker-compose.yml config
```

Check running containers:

```sh
docker compose -f srcs/docker-compose.yml ps
```

Expected result:

- `nginx` is up and publishes `443`
- `wordpress` is up and exposes only internal `9000`
- `mariadb` is up and exposes only internal `3306`

Check HTTPS:

```sh
curl -k -I https://belinore.42.fr
```

```sh
curl -k -v https://belinore.42.fr
```

Check nginx loaded configuration:

```sh
docker compose -f srcs/docker-compose.yml exec nginx nginx -T
```

Important lines:

```text
listen 443 ssl;
ssl_protocols TLSv1.2 TLSv1.3;
ssl_certificate /run/secrets/ssl_certificate;
ssl_certificate_key /run/secrets/tls_private_key;
fastcgi_pass wordpress:9000;
```

Check MariaDB is connected to the WordPress container:

```sh
docker compose -f srcs/docker-compose.yml exec wordpress \
	nc -zv mariadb 3306
```

For an interactive database check, use the actual database user and password from the secret file.

```sh
docker compose -f srcs/docker-compose.yml exec mariadb sh
```

Useful SQL:

```sql
SELECT CURRENT_USER(), VERSION(), DATABASE();
SHOW TABLES;
SHOW GRANTS FOR CURRENT_USER();
```

Check WordPress users:

```sh
docker compose -f srcs/docker-compose.yml exec wordpress \
	wp user list --path=/var/www/html --allow-root
```

There should be two users:

- one administrator
- one author

The administrator username must not contain `admin` or `administrator`.

- set Compose volume devices to `/home/belinore/data/mariadb` and `/home/belinore/data/wordpress`
- optionally create password secret files in `secrets/`
- let `make` generate any missing secret and TLS files
- confirm `srcs/.env` contains no passwords
- confirm `/etc/hosts` maps `belinore.42.fr` to the VM/local IP
- run `make fclean`
- run `make up`
- check `make ps`
- check `curl -k -I https://belinore.42.fr`
- verify no generated data, secrets, private keys, certificates, or `.DS_Store` files are staged in git

## Notes On Persistence

The MariaDB container initializes the database only when the MariaDB data directory is empty.

The WordPress container installs WordPress only when WordPress is not already installed.

This means restarting containers does not normally overwrite existing site data. To rebuild from scratch, remove the volumes, local data, and generated secrets with:

```sh
make fclean_data
```
